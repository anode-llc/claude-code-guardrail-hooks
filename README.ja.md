# claude-code-guardrail-hooks

**English version: [README.md](README.md)**

AI エージェントが守ってくれない「書かれたルール」を、エージェントが協力しようがしまいが必ず走る「機構」に変えるための、Hook テンプレート集です。

AI エージェントに指示書（`CLAUDE.md` やハウススタイルの文書）を渡しても、エージェントがルールを破り続ける——これは欠陥ではありません。自然言語の指示への追従はそもそも確率的で、セッションが長くなるほどルールは後景に退きます。処方は、指示書を分厚くすることではありません。破られると実害が出るルールを、「お願い」の世界から「機械が必ず実行する」の世界へ移すことです。Claude Code では、その機構が **hook** です。hook は、エージェントの作業の節目——書き込みの前、書き込みの後、セッションの開始時、セッションの終了時——に、エージェントの判断を経由せずに必ず走るスクリプトです。

ただし機構は強力です。強すぎる統制は現場を固めます。正当な作業まで止め、エージェントは同じ壁に何度もぶつかり、人間が張り付いて解除する羽目になる。だから狙いは「すべてを機構化する」ことではありません。狙いは、それぞれの統制の強さを、そのルールを破ったときの実害に見合わせることです。このリポジトリは、ルールを振り分けるための4段階と、各段階の骨組みとなる hook を1つずつ提供します。

## 統制の4段階 決定表

指示書を1行ずつ見て、各ルールに3つの問いを立ててください。

1. **違反コスト** — 破られたときの実害はどれくらいか？
2. **誤検知リスク** — 検査が正当な作業まで止めてしまう可能性はどれくらいか？
3. **人間による上書き** — 人間が意図的にルールを曲げる必要がある場面はあるか？

そのうえで、各ルールを4段階のどれかに振り分けます。何でもかんでも DENY に寄せないこと。強い統制は、実害の大きい行にだけ使います。

| 段階 | いつ使うか（判断軸） | Claude Code のイベント | このリポジトリの hook |
|---|---|---|---|
| **(1) Prevent（予防）** | 違反を起こさせない方が安い／毎回思い出させたい前提 | `SessionStart` — 作業の開始時に前提を注入する | `hooks/session-start-inject.sh` |
| **(2) Deny（禁止・ハード統制）** | 破ると実害が出る **かつ** 検出パターンを絞れる | `PreToolUse` — 書き込みの直前に拒否する | `hooks/pre-write-deny.sh` |
| **(3) Warn（警告・ソフト統制）** | 1件あたりの実害は小さいが放置で腐る／これで止めたくはない | `PostToolUse` — 書き込みの後に警告する（止めない） | `hooks/post-write-warn.sh` |
| **(4) Remind（催促・不作為の捕捉）** | 「やらなかった」（書き忘れた記録）を捕まえたい | `Stop` / `SubagentStop` — 作業の終了時に促す | `hooks/stop-remind.sh` |

一言ずつまとめると、こうなります。

- **(1) Prevent（予防）** — そもそも違反を起こさせない。あとで違反を捕まえるより、前提を先に渡す方が安い。新人に毎回オンボーディング資料を渡すのに似た、教育コストの前払いです。
- **(2) Deny（禁止）** — いちばん強い段階。破られると実害が出て、**かつ** 検出パターンが誤検知を避けられるほど絞れるルールだけに使います。強力なぶん危険なので、2つの安全装置を同梱しています（後述）。
- **(3) Warn（警告）** — 止めるほどではないが、放置すると腐るもの向け。書き込みは通し、あとで検査して警告だけ出す。止めないので、誤検知しても現場は固まりません。
- **(4) Remind（催促）** — これは毛色が違い、**不作為** を捕まえます。DENY hook は書き込みを検査するので、そもそも書かれなかった必要な記録には原理的に気づけません。この段階はセッションの終了時に促します——議論だけで終わったセッションを煩わせないよう、あくまで柔らかく。

## インストール

hook は `$CLAUDE_PROJECT_DIR` を読むので、リポジトリ内のどこからでも動きます。

```bash
# 1. hook を自分のプロジェクトにコピーする。
git clone https://github.com/anode-llc/claude-code-guardrail-hooks
cp -r claude-code-guardrail-hooks/hooks ./hooks && chmod +x ./hooks/*.sh

# 2. まだ .claude/settings.json が無ければ、example をそのままコピーする。
#    すでにある場合は、settings.example.json の "hooks" ブロックを
#    手作業でマージする（既存の設定は残す — ファイルを上書きしないこと）。
[ -f .claude/settings.json ] || cp claude-code-guardrail-hooks/settings.example.json .claude/settings.json
```

各スクリプトは **骨組み** です。汎用のプレースホルダのルールでそのまま動きますが、`PLACEHOLDER` と印を付けた検出ロジックは、自分のルールに置き換える前提です。あえて **含めていない** ものがあります——環境固有の例外の許可リスト（「違反」に見えて実は問題ない箇所の一覧）です。そのリストはあなたのルールに固有なので、骨組みは枠だけを残し、中身はあなたが埋める形です。

`bash`・`jq`・`python3` が `PATH` にあることを前提とします。

## 各スクリプト

### `hooks/session-start-inject.sh` — (1) Prevent（予防）

`SessionStart` で走ります。`hookSpecificOutput.additionalContext` にテキストを返すと、Claude Code がそれをセッションの先頭に前置きします。毎セッション思い出させたい前提（直近の作業ログの末尾、有効な制約、ハウスルールへのポインタ）を `build_context()` に埋めてください。短く保つこと——このテキストは毎セッション消費されます。注入するものが無ければ、何も出力しません。

### `hooks/pre-write-deny.sh` — (2) Deny（禁止）

`Edit`/`Write` に対して `PreToolUse` で走ります。保留中の書き込みを stdin から読み、それが違反を持ち込む場合は、書き込みを止める構造化された **deny** の判断を返します。`find_violations()` に自分の検出を埋めてください。2つの安全装置を同梱し、5秒のタイムアウトを想定した設計です（後述）。

### `hooks/post-write-warn.sh` — (3) Warn（警告）

`Edit`/`Write` に対して `PostToolUse` で走ります。書き込みはすでに済んでいるので、ファイルをディスクから読み、走査し、フラグ対象のパターンを見つけたら警告を出します。決して止めません。走査には自分の検出を埋めてください（例: 英語で公開するリポジトリに残った英語以外のコメント、消し忘れのマーカー、秘密情報らしき文字列）。

### `hooks/stop-remind.sh` — (4) Remind（催促）

`Stop` / `SubagentStop` で走ります。期待される成果物が欠けていないかを確認し、そのセッションで実際に作業が行われていれば（トランスクリプトを読み、議論だけのセッションはスキップします）、成果物を作るよう促します。ハードには止めません。

## DENY hook の安全装置

ハードな禁止は、いちばん効く段階であると同時に、いちばん現場を固めやすい段階です。DENY hook を常設できるものにするには、2つの安全装置が要ります。これを欠いた deny hook は、遅かれ早かれ外されます。

### (a) 新しい違反だけを見る（誤爆防止）

素朴に作ると、ファイル全体を検査し、どこか1単位でもルール違反があれば止める、という機構になります。これは即座に破綻します。ルールを導入する前に書かれた既存の違反がファイルに残っていると、そのファイルの——たとえ無関係な箇所への——今後のあらゆる編集が、その古い無関係な違反を理由に止まる。ツールがデッドロックします。

処方はこうです。編集操作には、直す前のテキスト（`old_string`）と直した後のテキスト（`new_string`）の **両方** が入っています。新しいテキストが持ち込み、かつ古いテキストには無かった違反だけを検査する。既存の負債は素通しする。古い負債の掃除を毎回の書き込みで強要しない、ということです。この差分は `pre-write-deny.sh` の `find_violations()` に実装してあります。

### (b) 緊急バイパス（逃げ道）

機械的な禁止は、正しく効いているときほど、人間の正当な「今回だけ」を封じます。たとえば古い記録の移行では、途中で一時的に無効な状態を通る必要があるかもしれません。機械が一切の例外を許さなければ、その正当な作業は進みません。明示的なバイパスをちょうど1つ用意してください——1タスクのあいだだけ検査を無効にする環境変数です。

```bash
SKIP_GUARDRAIL_CHECK=1 claude   # この実行のあいだ deny hook を無効にする
```

恒常的に立てたままにするのは、それ自体が規程違反です。しかし、人間が「このタスクでは意図的にオフにする」と判断したときの逃げ道として、これは必ず存在しなければなりません。（このテンプレートの元になった内部運用では、ルール固有の名前 `KOVITO_SKIP_SUMMARY_CHECK` を使っています。公開テンプレートでは汎用の `SKIP_GUARDRAIL_CHECK` を使っています——自分のルールに合わせて改名してください。）

### 5秒のタイムアウト

`settings.json` で deny hook に短いタイムアウト（5秒）を配線してください（`settings.example.json` 参照）。検査が万一ハングしても、タイムアウトがそれを握り潰し、書き込みは進みます。これは、統制そのものが壊れて全作業をデッドロックさせる事故への保険です——壊れたガードレールは、現場を固める（fail closed）のではなく、開いて通す（fail open）べきです。

## 技術ノート: なぜ非ゼロ終了ではなく `permissionDecision: "deny"` なのか

Claude Code を `bypassPermissions`（「いちいち確認してこなくていい」モード）で動かしていると、落とし穴があります。素朴なやり方で「止める」hook——非ゼロのステータスで終了する hook——は、書き込みを確実には止めません。bypassPermissions 下では、書き込みが素通りしてしまうことがあります。確実に止めるには、hook が構造化された deny の判断を stdout に返し、終了コード `0` で終わる必要があります。

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "…なぜ止めたのか…"
  }
}
```

経営の言葉に翻訳するとこうなります——統制は「入れたつもり」が最も危ない。運用のモード次第で、入れたはずの統制が実際には何も止めていないことがある。だから測るべきは「統制を入れたか」ではなく「実際に止まるか」です。自分が運用しているモードで、deny hook が本当に止まるかをテストしてください。
